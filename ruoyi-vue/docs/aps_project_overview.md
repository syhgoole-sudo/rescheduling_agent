# 面向插单扰动的车间动态重调度智能决策系统

## 1. 项目名称

面向插单扰动的车间动态重调度智能决策系统。

## 2. 项目背景

离散制造车间在执行生产计划时，经常会遇到高优先级插单。插单会占用原计划中的设备能力和时间窗口，导致原有订单延期、设备负荷变化、任务顺序调整等问题。传统人工调整方式依赖调度员经验，响应慢、影响范围不透明、方案取舍缺少量化依据。

本项目面向“插单扰动”这一典型动态调度场景，构建一个可运行的 APS 原型系统，实现从初始排产、插单影响分析、局部重调度、方案对比到调度员人工确认的闭环。

## 3. 业务问题

系统重点解决以下问题：

- 插单到来后，如何判断它会影响哪些设备、任务和订单。
- 如何在不全局推翻原计划的前提下，形成局部重调度方案。
- 如何保存原方案与新方案，支持方案追溯。
- 如何用 KPI 和甘特图帮助调度员判断新方案是否可采用。
- 如何用解释报告说明策略选择、收益、代价和建议。

## 4. 系统目标

- 建立车间基础数据、订单、工艺路线和设备资源的维护能力。
- 生成一个可执行的初始调度方案。
- 对高优先级插单进行影响范围分析。
- 推荐局部重调度策略，并生成新的 RESCHEDULE 方案。
- 支持新旧方案 KPI 对比和甘特图对比。
- 支持调度员采用或拒绝重调度方案。
- 使用规则模板生成解释报告，辅助调度员理解方案。

## 5. 技术架构

技术栈：

- 后端：若依 Spring Boot
- 前端：若依 Vue
- 数据库：MySQL
- 缓存与登录支撑：Redis
- 算法服务：Python FastAPI
- 可视化：ECharts
- 调度算法：规则调度算法、局部重调度算法
- 解释模块：规则模板式解释报告

架构分工：

- Vue 前端负责基础 CRUD、按钮操作、甘特图展示、对比弹窗和解释报告展示。
- Java 后端负责业务数据管理、方案版本管理、权限控制、接口编排和结果落库。
- MySQL 保存基础数据、订单、调度方案、任务明细和插单事件。
- Redis 支撑若依登录、权限和缓存。
- Python FastAPI 负责初始调度和局部重调度计算。
- 解释报告模块只解释方案，不参与排产计算。

## 6. 核心模块

- 基础数据管理：产品、工艺路线工序、设备组、设备。
- 订单管理：普通订单和插单订单统一维护。
- 初始调度模块：调用 Python 服务生成 INITIAL 方案。
- 调度方案管理：保存方案主表、方案状态和版本关系。
- 调度任务管理：保存任务明细，并支持按方案查询。
- 甘特图展示：按设备维度展示任务时间条。
- 插单事件管理：创建插单事件，记录原方案和新方案。
- 影响范围分析：识别受影响任务、订单、设备和影响等级。
- 策略推荐：根据影响等级推荐局部重调度策略。
- 局部重调度：生成 RESCHEDULE 新方案。
- KPI 对比：计算延期、完工时间、变更任务等指标。
- 新旧甘特图对比：直观展示插单任务、变更任务和冻结任务。
- 方案确认：调度员采用或拒绝重调度方案。
- 解释报告：生成面向调度员的方案说明。

## 7. 完整业务流程

1. 维护产品、工艺路线、设备组和设备。
2. 录入普通订单和插单订单。
3. 调度员在调度方案页面点击“生成初始调度”。
4. Java 后端查询订单、工艺路线和设备资源，组装请求调用 Python FastAPI。
5. Python 返回初始调度任务和 KPI。
6. Java 后端保存 INITIAL 方案和调度任务明细。
7. 前端查看任务明细和单方案甘特图。
8. 调度员创建插单事件并触发影响范围分析。
9. 系统保存 impact_json，展示影响等级、受影响任务、订单和设备。
10. 调度员点击“推荐策略”，系统推荐局部重调度策略。
11. 调度员点击“生成局部重调度方案”。
12. Java 后端根据冻结任务、可调整任务和插单任务调用 Python 局部重调度接口。
13. Python 返回完整新方案任务列表和 KPI。
14. Java 保存 RESCHEDULE 方案和任务明细。
15. 调度员查看新旧 KPI 对比和甘特图对比。
16. 系统生成规则模板式解释报告。
17. 调度员人工决定采用或拒绝新方案。

## 9. 系统边界

当前 MVP 明确不包含：

- 设备故障扰动。
- MES、ERP、WMS 等外部系统集成。
- 多扰动叠加处理。
- 真实车间执行反馈。
- 历史案例库。
- 高级元启发式优化算法。
- 外部大模型 API 接入。
- AI 自动排产。

AI/Agent 边界：

- 不直接生成排产结果。
- 不直接分配设备。
- 不绕过工艺、设备、时间约束。
- 只解释影响范围、策略原因、KPI 差异和方案取舍。

## 10. 后续可扩展方向

- 设备故障、交期提前、加工时间延误等更多扰动类型。
- 多算法策略池，例如规则算法、遗传算法、禁忌搜索、强化学习调度。
- 调度方案版本树和确认记录表。
- MES/ERP 数据接口。
- 设备日历、班次、维护窗口和工装约束。
- 多目标优化：交期、换型、能耗、设备利用率。
- 真实 AI Agent 接入，用于策略解释、自然语言问答和报告生成。
- 执行跟踪与计划偏差预警。

## 11. 半导体可重入场景适配说明

本系统当前以通用车间动态调度为实现基础，但其核心对象可以映射到半导体可重入制造场景：

- `aps_order` 可解释为 lot / wafer lot。
- `aps_product` 可解释为产品类型或工艺族。
- `aps_route_operation` 可解释为工艺路线 step。
- `equipment_group` 可解释为 tool group / 设备族。
- `equipment` 可解释为具体 tool / 机台。
- `schedule_task` 可解释为 lot-step 加工任务。
- `insert_event` 可解释为插单 lot 扰动事件。

当前系统可以表达简化的可重入路线：同一个产品的工艺路线中，同一设备组可以在不同 `process_seq` 上出现多次。例如 demo 数据中的 `P-B` 路线包含 `CUT -> CNC -> HEAT -> CNC -> INSPECT`，其中 `CNC` 设备组重复出现，可用于模拟 lot 多次回到同一 tool group 的 reentrant flow。

当前 MVP 重点验证半导体插单重调度中的共性闭环：

1. 插单 lot 到达。
2. 根据 route steps 识别所需 tool groups。
3. 在当前 ACTIVE 方案中识别同 tool group、释放时间之后的未冻结 lot-step。
4. 形成受影响任务集合。
5. 冻结未受影响任务。
6. 重排受影响 lot-step 和插单 lot steps。
7. 通过 KPI 和甘特图对比辅助调度员决策。

当前系统尚未显式建模 Q-time、recipe、mask、batch、setup、tool qualification、layer、wafer 数量等半导体特有约束。后续可在不推翻现有闭环的基础上，逐步扩展 `aps_lot`、recipe、layer、Q-time、设备资格和批处理等模型。
