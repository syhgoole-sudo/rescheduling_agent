# 简历与面试描述

## 1. 简历项目名称

面向插单扰动的车间动态重调度智能决策系统

## 2. 一段式项目描述

基于若依 Spring Boot、Vue、MySQL、Redis 和 Python FastAPI 开发车间 APS 动态重调度原型系统，面向高优先级插单扰动场景，实现基础数据维护、初始调度、插单影响分析、局部重调度策略推荐、新方案生成、KPI 对比、甘特图可视化和规则模板式解释报告。系统中 AI/Agent 模块不直接排产，只负责解释影响范围、策略原因、KPI 差异和方案取舍，调度结果由规则调度算法和局部重调度算法生成。

## 3. 简历 Bullet

- 基于若依 Spring Boot + Vue 搭建 APS 原型系统，设计产品、工艺路线、设备、订单、调度方案、任务明细和插单事件等核心数据模型，支持基础 CRUD 和方案版本追溯。
- 使用 Python FastAPI 封装规则调度服务，实现普通订单初始排程和插单场景下的局部重调度，Java 后端通过 HTTP 调用算法服务并保存完整方案任务。
- 设计插单影响分析逻辑，根据插单工艺路线、设备组、释放时间和当前 ACTIVE 方案识别受影响任务、订单、设备和影响等级。
- 实现 RESCHEDULE 新方案与 INITIAL 原方案的 KPI 对比和 ECharts 甘特图对比，支持变更任务、插单任务、冻结任务的可视化区分。
- 实现调度员采用/拒绝重调度方案的人工决策闭环，并通过规则模板生成解释报告，说明方案收益、代价和采用建议。

## 4. 面试讲解版本

这个项目是一个面向制造车间插单扰动的 APS 动态重调度原型。我的目标不是做一个复杂的商业 APS，而是把一个完整的业务闭环做通：先维护产品、工艺路线、设备和订单，然后生成初始调度方案；当出现高优先级插单时，系统先分析它会影响哪些设备组、任务和订单，再推荐局部重调度策略，最后调用 Python 算法服务生成新的 RESCHEDULE 方案。

系统架构上，我用若依 Spring Boot 做业务后端，用 Vue 做前端页面，用 MySQL 保存方案和任务，用 Redis 支撑若依登录，用 Python FastAPI 做调度算法服务。Java 后端负责业务编排和方案落库，Python 只负责算法计算，不直接访问数据库。

调度算法第一版采用规则算法。初始调度按照优先级、交期和释放时间排序，每个订单按工艺路线顺序加工，并选择设备组中最早可用的设备。局部重调度时，未受影响任务冻结，受影响任务和插单任务重新排程，插单任务优先。

可视化方面，我实现了单方案甘特图和新旧甘特图对比，可以看到每台设备的任务时间条，也能区分插单任务、变更任务和冻结任务。决策方面，我实现了 KPI 对比，包括延期订单数、总延期时间、最大延期、makespan、变更任务比例等。最后调度员可以采用或拒绝新方案。

AI/Agent 部分我没有让它直接排产，而是用规则模板生成解释报告，解释插单影响、策略选择、KPI 变化、方案收益和方案代价。这符合制造系统中调度结果必须可追溯、可验证的要求。

## 5. STAR 版本

Situation：

车间生产计划执行过程中经常遇到高优先级插单，人工调整排产方案时难以及时判断影响范围，也缺少量化的方案对比依据。

Task：

设计并实现一个 APS 动态重调度原型系统，聚焦插单扰动场景，完成从初始调度到插单分析、局部重调度、方案对比和人工确认的闭环。

Action：

- 使用若依 Spring Boot + Vue 搭建业务系统，设计 MySQL 表结构并生成基础 CRUD。
- 使用 Python FastAPI 实现初始调度和局部重调度规则算法。
- 在 Java 后端实现算法服务调用、方案保存、任务保存和状态流转。
- 实现插单影响分析、策略推荐、KPI 对比、甘特图展示和解释报告。
- 保持 AI/Agent 边界清晰，只做解释和辅助决策，不直接生成排产方案。

Result：

完成一个可运行的 APS MVP 原型，能够演示普通订单初始排程、高优先级插单影响分析、局部重调度新方案生成、新旧方案 KPI 与甘特图对比，以及调度员采用/拒绝方案的完整流程。

## 6. 项目亮点

- 业务闭环完整：覆盖初始调度、扰动分析、策略推荐、重调度、对比和确认。
- 架构分层清晰：Java 管业务，Python 管算法，MySQL 管持久化，Vue 管交互展示。
- 制造业务贴近真实：围绕订单、工艺路线、设备组、设备、任务、方案版本展开。
- 方案可追溯：INITIAL 和 RESCHEDULE 方案通过 `source_plan_id` 建立关系。
- 可视化直观：使用甘特图展示设备维度任务安排和新旧方案差异。
- AI 边界合理：不宣称 AI 自动排产，而是用于解释和辅助调度员决策。

## 10. 半导体可重入版本项目描述

基于若依 Spring Boot、Vue、MySQL 和 Python FastAPI 开发面向半导体可重入制造场景的插单重调度原型系统，将订单抽象为 lot，将工艺路线工序抽象为 route step，将设备组抽象为 tool group，将设备抽象为 tool，通过同一产品路线中重复出现同一 tool group 表达 reentrant flow。系统实现插单 lot 影响范围分析、受影响 lot-step 识别、局部重调度策略推荐、RESCHEDULE 新方案生成、KPI 对比、新旧甘特图对比和规则模板式解释报告。AI/Agent 只负责解释影响、策略、KPI 和方案取舍，不直接生成排产结果。

## 11. 半导体方向面试讲解版本

如果从半导体可重入调度角度讲，这个项目可以理解为一个 hot lot 插入后的局部重调度决策支持原型。系统底层仍使用通用 APS 数据模型，但做了半导体语义映射：`aps_order` 对应 lot，`aps_route_operation` 对应 route step，`equipment_group` 对应 tool group，`equipment` 对应 tool，`schedule_task` 对应 lot-step 加工任务。

可重入特征通过工艺路线表达：同一个产品在不同 step 上可以多次经过同一设备组。例如 demo 中 `P-B` 的路线是 `CUT -> CNC -> HEAT -> CNC -> INSPECT`，其中 `CNC` 重复出现，可类比半导体流程中 lot 多次回到同一类设备族。

当插单 lot 到达时，系统根据该 lot 的 route steps 找到所需 tool groups，再从当前 ACTIVE 调度方案中识别同 tool group、release time 之后且未冻结的 lot-step 任务，形成受影响集合。随后系统冻结未受影响任务，对受影响 lot-step 和插单 lot steps 做局部重排，生成 RESCHEDULE 新方案。

项目重点不是宣称已经覆盖真实 Fab 的全部复杂约束，而是验证半导体插单重调度中的关键闭环：影响范围识别、局部重调度、方案版本追溯、KPI 对比、甘特图对比和调度员决策。当前还没有显式建模 Q-time、recipe、mask、batch、tool qualification 等半导体特有约束，这些是后续扩展方向。
